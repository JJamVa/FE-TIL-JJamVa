# /til-update — 블로그 포스트 업데이트 및 수정

대상 파일: **$ARGUMENTS**

`/til-review`와 달리 이 커맨드는 **원본 파일을 직접 수정**합니다. 파괴적 작업이므로 모든 단계를 **상세히, 꼼꼼히, 자세히** 수행해 실수를 철저히 방지합니다.

## 🔍 철저성 원칙 (Thoroughness Mandate)

- **훑어보기 금지** — 전체 파일을 한 문단도 빠뜨리지 말고 자세히 검토
- **추측 수정 금지** — 모든 변경은 권위 있는 출처(MDN / 공식 docs / RFC / W3C / Can I Use)로 뒷받침
- **승인 없는 수정 금지** — 발견 사항을 먼저 제시하고 사용자 명시적 승인을 받은 뒤에만 Edit 실행
- **self-review 필수** — 수정 완료 후 Read로 다시 읽어 의도대로 반영됐는지 2회 이상 꼼꼼히 확인
- **Write 금지** — 원본 덮어쓰기 불가, 반드시 Edit tool로 섹션 단위 수정

## 절차

### 1. 대상 파일을 상세히 읽기
- Read tool로 `$ARGUMENTS` **전체**를 처음부터 끝까지 상세히 읽기 (limit 없이)
- frontmatter·본문·코드 블록·인용·이미지·외부 링크까지 하나도 빠뜨리지 말고 꼼꼼히 스캔
- 모든 기술적 주장·API 시그니처·버전 표기·URL을 내부적으로 자세히 목록화

### 2. frontend-blog-verifier 에이전트로 심층 검증 (자세히)
Agent tool 호출 시 다음을 상세히 지시:
- "이미 작성된 포스트입니다. **자동 수정하지 마세요.** 검증 + 수정안 제시까지만 수행합니다."
- 모든 기술적 주장을 하나도 빠뜨리지 말고 상세히 추출
- 각 주장을 최신 공식 문서와 자세히 대조해 아래로 분류:
  - ⛔ **오래된 정보** — 버전 변경, deprecated, 동작 변경
  - ❌ **오류** — 처음부터 잘못된 내용
  - 🔗 **죽은/변경된 링크** — 404, redirect, 이동된 URL
  - 📅 **버전 미명시** — 현재성을 잃은 주장
  - ⚠️ **보완 필요** — 부분적 사실, 맥락 누락
- 코드 예제를 문법·API 사용·최신 베스트 프랙티스 관점에서 꼼꼼히 검증
- 각 발견 항목에 대해 **구체적 수정안** 제시 (원문 → 수정안 + 출처 URL)

### 3. 발견 사항을 먼저 사용자에게 자세히 제시
**원본을 즉시 수정하지 말 것.** 다음 요약을 먼저 보여줌:

```
## 발견 사항 요약
총 N개 항목 — 중대 N / 보통 N / 경미 N

### 중대 (즉시 수정 권장)
1. [섹션명] "원문 일부" → "수정안" (출처: URL)
2. ...

### 보통
...

### 경미
...
```

**사용자 명시적 승인 대기** ("진행해줘" / "OK" / "수정해줘" 등). 승인 전까지 Edit 호출 금지.

### 4. 승인 후 Edit tool로 원본을 단계적·상세히 수정
- **Write tool로 전체 덮어쓰기 절대 금지** — 실수 시 복구 어려움
- Edit tool로 **섹션 단위**로 꼼꼼히 수정
- 각 Edit의 `old_string`은 충분한 컨텍스트를 포함해 오·수정 방지
- 블로그 스타일을 자세히 유지:
  - 제목·본문 한국어, 코드·기술 용어 영어
  - `:::note` admonition 패턴 유지
  - 코드 블록 언어 힌트(```js, ```html 등) 필수
  - 인라인 코드는 백틱, 리스트는 `-` 하이픈
- 수정 후 반드시 Read tool로 다시 읽어 의도대로 반영됐는지 상세히 검증

### 5. frontmatter 꼼꼼히 업데이트
- `last_update: YYYY-MM-DD` 필드가 있으면 오늘 날짜로 갱신
- 없으면 frontmatter 상단에 `last_update: <오늘>` 추가 권장
- 오늘 날짜: 현재 시스템 날짜 사용

### 6. 변경 로그를 상세히 저장
경로: `review/<원본파일명 확장자 제외>-update-<YYYY-MM-DD>.md`
예: `docs/frontend/8-React/13-useEffect.md` 업데이트 → `review/13-useEffect-update-2026-04-19.md`

구조(모든 항목을 자세히 채울 것, 빈 필드 금지):

```markdown
# 🔄 업데이트 로그: <원본 제목>

- 대상: `<원본 경로>`
- 업데이트 일자: <YYYY-MM-DD>
- 수행 에이전트: frontend-blog-verifier

## 📊 요약
- 수정 항목: N개
- 분류: ⛔오래됨 N / ❌오류 N / 🔗링크 N / 📅버전 N / ⚠️보완 N
- 위험도: 중대 N / 보통 N / 경미 N

## 🔧 변경 내역

### 1. [섹션명 / 줄 번호 대략]
- **분류**: ⛔오래됨 / ❌오류 / 🔗죽은 링크 / 📅버전 미명시 / ⚠️보완
- **기존**: "원문 인용"
- **변경**: "수정 결과"
- **사유**: <왜 잘못·오래되었는지 상세히>
- **출처**: <URL>
- **위험도**: 중대 / 보통 / 경미

(발견 항목 수만큼 꼼꼼히 반복)

## 📚 참고한 권위 자료
- <URL 리스트 — 누락 없이 자세히>

## 🔁 재검토 권장 시점
- <주제가 빠르게 변하는 영역이면 재검토 주기 제안 (예: "6개월 후")>
- <특정 버전/스펙 업데이트를 트리거로 명시>
```

### 7. 결과 상세히 알림
- 수정된 원본 경로
- changelog 파일 경로
- 수정 규모 요약 (N개 항목, 위험도 분포)
- 추가 정밀 검증을 원하면 `/til-review <원본>` 실행 안내
- 전체 변경을 Git으로 한 번에 커밋할지 여부를 사용자에게 확인 안내

## 엄격한 제약
- **사용자 승인 없이 수정 금지**
- **Write tool로 원본 덮어쓰기 금지** — Edit tool만 사용, 섹션 단위
- **증거 없는 변경 금지** — 모든 수정은 권위 있는 출처 동반
- **스타일·취향 영역 수정 지양** — 오직 사실성·최신성·오류·죽은 링크에 집중
- **수정 후 재확인 필수** — Read tool로 결과를 자세히 검증
- 한 번의 세션에서 지나치게 많은 수정이 발생하면(20개 이상) 사용자에게 분할 처리 제안

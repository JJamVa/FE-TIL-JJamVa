# /til-write — 블로그 본문 작성

기획서 파일: **$ARGUMENTS**

## 🔍 철저성 원칙 (Thoroughness Mandate)

- 기획서를 **처음부터 끝까지 자세히** 파싱. 한 섹션도 건너뛰지 말 것.
- 에이전트에 전달할 컨텍스트는 기획서 전체 + 스타일 규칙 + 제약을 상세히 포함
- 본문 저장 후 Read로 다시 읽어 의도대로 쓰였는지 꼼꼼히 확인 (최소 1회)
- 사실 확인이 어려운 주장은 얼버무리지 말고 **제거** 또는 **출처 명시**
- 기획서 범위를 벗어난 주제 추가는 엄격 금지 (scope creep 차단)

## 절차

### 1. 기획서 상세히 파싱
- Read tool로 `$ARGUMENTS` **전체** 읽기 (limit 없이)
- 추출할 정보를 하나도 빠뜨리지 말고 자세히 수집:
  - "📍 배치" → 본문 저장 대상 경로
  - "🎯 대상 독자" → 글의 난이도·톤
  - "📑 목차 초안" → 섹션 구조
  - "🔍 섹션별 상세" → 각 섹션 내용 지침
  - "📚 참고 자료" → 인용할 권위 있는 출처 전부
  - "⚠️ 검증 필요 항목" → 작성 시 특히 주의할 주장
  - "🚫 범위 밖" → 절대 다루지 말 것

### 2. frontend-blog-verifier 에이전트로 본문 꼼꼼히 작성
Agent tool 호출 시 다음을 상세히 지시:

- 기획서 전체 내용을 컨텍스트로 전달
- "기획은 완료된 상태입니다. **기획서 범위를 벗어나지 말고** Phase 1(작성) + Phase 2(검증)을 상세히, 꼼꼼히 수행하세요."
- 아래 **블로그 스타일 규칙**을 자세히 준수하라고 명시
- "⚠️ 검증 필요 항목"의 모든 주장에 대해 공식 출처와 대조한 뒤 서술

### 3. 블로그 스타일 규칙 (자세히 준수)
본 프로젝트 기존 포스트 패턴을 꼼꼼히 따를 것:

- 제목: `# 주제명` (한 줄, 한국어)
- 섹션 제목: `## 섹션명`
- 하위 섹션: `### 하위명`
- 개념 설명·Tip 박스: `:::note` admonition 사용
  ```
  :::note
  개념 설명 ...
  :::
  ```
- 코드 블록은 **반드시 언어 힌트** (```js, ```html, ```css, ```tsx 등)
- 인라인 코드·키워드는 백틱: `useState`, `Array.prototype.map`
- 한국어 본문 + 영어 코드·기술 용어
- 리스트는 `-` 하이픈 사용
- 긴 보충 설명은 `<details><summary>` 블록 가능
- 문장 끝은 일관되게 (평서체 / 해요체 중 하나로 통일)

### 4. 본문을 대상 경로에 상세히 저장
- 기획서의 "📍 배치 → 대상 경로"로 Write tool 저장
- 예: `docs/frontend/1-HTML/12-h태그.md`
- 해당 폴더/파일이 이미 존재하면 덮어쓰기 전에 **반드시 사용자에게 확인**
- 저장 후 Read로 다시 읽어 문법·admonition 태그·코드 블록이 제대로 들어갔는지 꼼꼼히 검증

### 5. 생성 단계 검증 보고서를 자세히 저장
에이전트의 Phase 2(검증) 결과를 별도 파일로:
- 경로: `review/<원본파일명 확장자 제외>-verification.md`
- 예: 본문이 `12-h태그.md`이면 → `review/12-h태그-verification.md`
- `review/` 폴더가 없으면 Write tool이 자동 생성
- 에이전트가 반환한 검증 결과(✅/⚠️/❌/❓)를 하나도 빠뜨리지 말고 상세히 수록

### 6. 결과를 상세히 알림
- 본문 경로
- 검증 보고서 경로
- Phase 2에서 ❌ 오류가 발견됐으면 개수와 심각도를 강조 (미수정 상태 명시)
- 추가 정밀 리뷰가 필요하면 `/til-review <본문경로>` 안내
- 오래된 내용 업데이트가 필요하면 `/til-update <본문경로>` 안내

## 엄격한 제약
- 기획서에 없는 주제·섹션 임의 추가 금지 (scope creep 엄격 차단)
- API·함수 이름 환각 금지 — 불확실하면 공식 문서 확인 후 작성 또는 제외
- 버전 정보가 필요한 주장은 반드시 버전 명시
- 권위 있는 출처로 추적 가능한 주장만 단정적으로 서술
- 검증에서 ❌ 오류 발견 시 해당 섹션을 반드시 재작성하거나 제거한 뒤 최종 저장
- 저장 후 재확인 필수 (Read tool로 꼼꼼히 검증)

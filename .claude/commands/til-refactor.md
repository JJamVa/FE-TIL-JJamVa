# /til-refactor — 전체 블로그 구조 리팩토링

## ⚠️ 위험도: 매우 높음

이 커맨드는 여러 파일을 **이동·수정·병합·분할·삭제**할 수 있는 **가장 파괴적인 작업**입니다. 반드시 2단계 프로토콜을 준수합니다:

- **Phase A (분석)**: 전체 블로그를 스캔하고 **로그만** 생성. 파일을 변경하지 않음.
- **Phase B (실행)**: Phase A의 로그를 사용자가 자세히 읽고 **명시적으로 승인한 항목만** 실행.

## 🔍 철저성 원칙 (Thoroughness Mandate)

- 모든 파일을 상세히, 꼼꼼히, 자세히 분석. 샘플링·건너뛰기 금지.
- 이동·수정 전 **반드시 사용자 승인 대기**.
- 각 변경에 구체적 근거(권위 있는 출처 / 분류 기준)를 자세히 제시.
- Phase B 실행 후 Read·git status로 재확인 필수 (최소 2회).
- **자동 커밋 금지** — 사용자가 확인한 뒤 수동 커밋.

---

## Phase A: 전체 스캔 및 로그 생성 (비파괴)

### A-1. 사전 안전 점검 (자세히)
1. `git status` 실행해 워킹 트리가 깨끗한지 확인
2. 미커밋 변경 사항이 있으면:
   - 사용자에게 알림: "작업 전에 커밋 또는 스태시를 권장합니다"
   - 그래도 진행할지 명시적 확인 대기
3. 현재 브랜치 표시 (main에서 직접 작업 시 주의 안내)

### A-2. 전체 문서를 상세히 수집
- Glob으로 `docs/**/*.md` 전체 파일 목록화 (**스캔 제외**: `.claude/`, `plan/`, `review/`, `node_modules/`, `build/`)
- 각 파일을 Read tool로 **처음부터 끝까지** 읽기 (limit 없이)
- 내부 메타데이터 목록화:
  - 경로 / 제목(`#`) / 섹션 헤더(`##`) / 주요 개념 bullet / 코드 예제 / 외부 링크

### A-3. 각 파일을 네 가지 축으로 꼼꼼히 분석

**축 1 — 배치 적합성**
- 현재 폴더/섹션이 주제에 적합한가?
- 숫자 prefix 순서가 논리적인가? (개념 → 심화 순서)
- 다른 섹션으로 이동이 더 나은가? (예: `3-JavaScript/`에 있지만 실제 React 내용)

**축 2 — 내용 정확성** (`frontend-blog-verifier` 에이전트 호출)
- 기술적 오류
- 오래된 정보 / deprecated API / 버전 누락
- 죽은 링크 / 이동된 URL
- 환각된 메서드·옵션

**축 3 — 중복·겹침** (`frontend-skill-explorer` 에이전트 호출)
- 같은 주제를 여러 파일이 다루는가?
- 두 파일을 병합하거나, 한쪽을 삭제/통합할 여지
- 한 파일이 너무 커서 분할이 필요한가? (7개 이상 대섹션)

**축 4 — 구조 일관성**
- 블로그 스타일 규칙 준수 여부:
  - `:::note` admonition 사용 패턴
  - 코드 블록 언어 힌트 유무
  - 한국어 본문 + 영어 코드
  - 제목 계층(`# > ## > ###`)
- frontmatter 유무 (있다면 일관성)

각 파일에 대한 발견 사항을 **권장 조치 유형**으로 분류:
- 🔀 **이동**: 다른 폴더로
- ✂️ **분할**: 2개 이상 파일로
- 🔗 **병합**: 다른 파일로 통합
- ❌ **내용 오류**: 수정 필요
- 🗑️ **삭제**: 중복 또는 무가치
- ✨ **구조 개선**: 스타일 조정
- ✅ **이상 없음**: 손대지 않음

### A-4. 로그 디렉토리·파일을 상세히 생성
기본 경로: `review/refactor-<YYYY-MM-DD>/`

생성 파일:

**1. `summary.md`** — 전체 요약 + 권장 조치 목록

```markdown

- 스캔 파일 수: N개
- 분석 에이전트: frontend-blog-verifier, frontend-skill-explorer

## 📊 발견 사항 요약
- 🔀 이동 권장: N개
- ✂️ 분할 권장: N개
- 🔗 병합 권장: N개
- ❌ 내용 오류: N개
- 🗑️ 삭제 권장: N개
- ✨ 구조 개선: N개
- ✅ 이상 없음: N개

## 📋 권장 조치 상세 (번호는 승인 시 사용)

- [S1] `docs/...` → 2개 파일로

  - 중복도: <자세히>
  - 병합 후 삭제할 파일:

### ❌ 내용 오류
- [E1] `docs/...` 에서 발견:
  - 원문: "..."
  - 문제: <상세히>
  - 수정안: "..."
  - 출처: <URL>

### 🗑️ 삭제 권장
- [D1] `docs/...` — 이유: <상세히>

- [F1] `docs/...` — 변경: <예: 코드 블록에 언어 힌트 추가>

2. **선택 승인**: 항목 번호 지정 (예: "M1, S1, E1,E2 실행")
3. **보류**: "중지" / "나중에"
```

**2. `per-file/<슬러그>.md`** — 각 파일별 상세 분석 (발견 사항 있는 파일에 한해 생성)


### A-5. 사용자에게 제시
- `summary.md` 경로 알림
- 발견 사항 개수 요약 한 줄 (예: "중대 3 / 보통 12 / 경미 5")
- "자세히 읽어본 후 실행 항목을 알려주세요" 메시지
- **Phase B는 승인 전까지 시작 금지**

## Phase B: 승인된 항목만 단계적 실행 (파괴적)

### B-1. 승인 범위를 꼼꼼히 파싱
사용자 응답을 자세히 해석:
- "모두 실행" → 전체
- "M1, S1, E1" → 지정 항목만
- "위험도 낮음만" → 해당 조건만
- 불명확하면 다시 질문 (추측 금지)

### B-2. 변경 유형별 안전 실행

**🔀 이동**
- `git mv` (가능하면) 또는 Bash `mv`로 이동해 Git 히스토리 보존
- 이동 후 **숫자 prefix 재정렬** 필요 여부 확인
- 내부 링크 자세히 스캔·업데이트:
  - Grep으로 `](./원경로` 또는 관련 참조 탐색
  - Edit tool로 각 링크를 섹션 단위 수정
- 이동 후 Read로 대상 경로 파일 존재 여부 재확인

**🔗 병합**
- 대상 파일에 Edit tool로 섹션 추가 (Write로 덮어쓰기 금지)
- 중복 문단은 제거하되 문체 차이는 유지
- 병합 완료 후 사용자 **재확인**하고 원본 삭제
- 삭제 전 원본 내용을 `changelog`에 백업

**✂️ 분할**
- 새 파일 Write tool로 생성
- 원본에서 해당 섹션 제거는 Edit tool
- 양쪽 파일 Read로 재확인
- 내부 링크 업데이트

**❌ 내용 수정**
- `/til-update`와 동일한 단계별 Edit
- Write tool로 전체 덮어쓰기 금지
- 수정 후 Read로 꼼꼼히 재확인

**🗑️ 삭제**
- 삭제 전 **반드시 2차 확인**
- `git rm` 또는 Bash `rm`
- 삭제 내용을 changelog에 전문 백업

**✨ 구조 개선**
- Edit tool로 섹션 단위 조정
- 스타일 규칙(admonition, 코드 블록 언어 힌트 등) 상세히 준수

### B-3. 실행 로그를 자세히 기록
경로: `review/refactor-<YYYY-MM-DD>/execution-log.md`

```markdown
# 🛠️ 리팩토링 실행 로그

- 실행 일시: <YYYY-MM-DD HH:MM>
- 승인된 항목: [M1, S1, E1, ...]

## ✅ 성공
- [M1] <원경로> → <새경로> — Git mv 완료
- [E1] <파일> — 3개 섹션 수정
...

## ⚠️ 경고
- [J1] — 병합 후 사용자 재확인 대기

## ❌ 실패 / 스킵
- [D1] — 사용자가 삭제 취소

## 🔗 변경된 내부 링크
- `<파일A>`의 `[...](./old.md)` → `[...](./new.md)`
```

### B-4. 결과를 상세히 알림
- 변경된 파일 목록
- 실패·스킵된 항목
- `git status`로 변경 사항 요약 출력
- 커밋 권장 안내 (예시 커밋 메시지 제시 가능, 자동 커밋은 금지)
- 추가 정밀 검증이 필요한 파일이면 `/til-review <경로>` 안내

## 엄격한 제약

- **사용자 승인 없이 Phase B 금지**
- **자동 커밋 금지** — 반드시 사용자가 확인 후 수동 커밋
- **일괄 처리 금지** — 변경 유형별로 단계적으로
- **Write로 원본 덮어쓰기 금지** — Edit tool 우선
- 각 단계 후 Read·git status로 꼼꼼히 재확인
- 20개 이상 변경 발생 시 사용자에게 **분할 처리 제안**
- 스캔 범위 제외: `.claude/`, `plan/`, `review/`, `node_modules/`, `build/`, `.docusaurus/`, `src/`, `static/`, `CLAUDE.md`, `README.md`
- Phase A 중단 시 `review/refactor-<날짜>/` 폴더는 보존 (다음 세션에서 재사용 가능)
- 주요 이동 시 사이드바 숫자 prefix 순서가 깨지지 않도록 자세히 검증
- 한 세션 내에서 실행 가능한 범위가 너무 크면 사용자에게 "이번엔 A-B만, 다음 세션에 C-D" 같은 분할 전략 제안

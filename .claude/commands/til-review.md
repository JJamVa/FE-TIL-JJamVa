# /til-review — 블로그 포스트 정밀 리뷰

대상 파일: **$ARGUMENTS**

## 🔍 철저성 원칙 (Thoroughness Mandate)

- 포스트를 **한 문단도 빠뜨리지 말고 자세히** 분석
- 모든 기술적 주장·코드 예제·버전 표기·링크를 꼼꼼히 검증
- 모든 ❌/⚠️ 지적은 **권위 있는 출처(URL)** 과 함께 제시. 근거 없는 지적 금지.
- 원본은 절대 수정하지 않음 — 리뷰 보고서만 자세히 생성
- self-review: 리뷰 보고서 완성 후 Read로 다시 읽어 누락·오기재가 없는지 꼼꼼히 검증

## 절차

### 1. 대상 파일을 상세히 읽기
Read tool로 `$ARGUMENTS` **전체**를 처음부터 끝까지 자세히 읽기 (limit 없이).

### 2. frontend-blog-verifier 에이전트 실행 (검증 전용 모드)
Agent tool로 호출하며 다음을 꼼꼼히 지시:

- "이미 작성된 포스트입니다. 본문을 **수정하거나 재작성하지 마세요**. 오직 Phase 2(검증)만 상세히 수행."
- 모든 기술적 주장을 하나도 빠뜨리지 말고 자세히 추출
- 각 주장을 권위 있는 자료(MDN / 공식 docs / RFC / W3C·WHATWG specs / Can I Use)와 **꼼꼼히** 대조
- 각 주장을 4가지로 분류 (근거 URL 필수):
  - ✅ 확인됨
  - ⚠️ 보완 필요 (부분적 사실 / 맥락 누락)
  - ❌ 오류 (공식 문서와 모순)
  - ❓ 검증 불가 (근거 없음)
- 코드 예제의 문법·API 사용·동작 설명도 자세히 검증
- 버전·브라우저 호환성 주장의 정확도를 상세히 확인

### 3. 리뷰 보고서를 꼼꼼히 생성
경로: `review/<원본파일명 확장자 제외>-review.md`
예: 대상이 `docs/frontend/8-React/13-useEffect.md`이면 → `review/13-useEffect-review.md`
`review/` 폴더가 없으면 Write tool이 자동 생성.

보고서 구조(**모든 섹션을 상세히 채울 것**):

```markdown
# 📋 리뷰 보고서: <원본 제목>

- 대상 파일: `<원본 경로>`
- 리뷰 날짜: <YYYY-MM-DD>
- 수행 에이전트: frontend-blog-verifier (Phase 2 전용)

## 📊 요약
- 총 기술적 주장: N개
- ✅ 확인됨: N개
- ⚠️ 보완 필요: N개
- ❌ 오류: N개
- ❓ 검증 불가: N개

## ✅ 확인된 주장
- <주장 전문 또는 요약>
  - 출처: <URL>

## ⚠️ 보완 필요
- <주장 전문 또는 요약>
  - 문제: <무엇이 누락·오해의 소지가 있는지 자세히>
  - 제안 수정: <구체적 문구 제안>
  - 출처: <URL>

## ❌ 오류 (수정 필수)
- <주장 전문 또는 요약>
  - 왜 틀렸는지: <상세 근거>
  - 정확한 정보: <사실>
  - 출처: <URL>
  - 원문 위치: <섹션명 / 인용 일부>

## ❓ 검증 불가
- <주장>
  - 권장 조치: 출처 추가 / 해당 문장 제거 / 더 약한 표현으로 완화

## 🔧 제안 수정 사항 (꼼꼼히)
- **원문**: "..."
- **수정안**: "..."
- **이유**: <상세히>

(오류 건수만큼 꼼꼼히 반복)

## 📚 참고한 권위 자료
- <URL 리스트 — 누락 없이 자세히>
```

### 4. 결과를 상세히 알림
- 리뷰 파일 경로
- 요약 통계 (✅ / ⚠️ / ❌ / ❓ 개수)
- ❌ 오류가 있으면 수정이 필요한 항목 수·심각도를 강조 안내
- **자동 수정은 하지 않음**을 명시하고, 수정을 원하면 `/til-update <원본>` 또는 수동 수정 안내

## 엄격한 제약
- **원본 파일 자동 수정 금지** — 리뷰 보고서만 상세히 생성
- 근거 없는 지적 금지 — 모든 ❌/⚠️는 권위 있는 출처와 함께
- 스타일·문장 구성에 대한 지적 지양 — 오직 사실 정확성·기술적 오류에 집중
- 리뷰 보고서 저장 후 Read로 꼼꼼히 검증해 누락·오기재 차단
